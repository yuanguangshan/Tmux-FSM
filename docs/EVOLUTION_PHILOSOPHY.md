# The Ascent to Kernel Sovereignty
## 从按键模拟器到无头内核的演进哲学

---

### 1. 速度不仅是性能，而是认知带宽  
**Latency Is Cognition**

在早期设计中，我们容忍了进程启动和 IPC 的开销。但在编辑领域，  
**50ms 的延迟并不是“慢”，而是对心流的结构性破坏。**

当架构从 CLI 演进为 Daemon，这并非一次性能优化，而是一次**认知层级的修复**：

- **零延迟不是目标，而是前提**
- 当工具的反应速度超越神经反射，工具本身才会消失
- 剩下的，只有意图与结果之间的直接映射

---

### 2. 环境无关性  
**Environment Agnosticism**

Vim 是伟大的，但它的哲学被封闭在 Vim 的 UI 边界之内。

现代开发者生活在 Shell、REPL、Notebook 与远程终端之间，但编辑能力却在环境切换中不断退化。

Weaver Core 的演进验证了一个判断：

> **编辑能力不应属于某个 UI，而应属于用户本身。**

通过将 `dw`、`ci"` 等语义从 Vim 抽离，并注入 Readline 与 ANSI 世界：

- 编辑逻辑获得了迁移性
- 用户的肌肉记忆获得了主权

**一致性不是便利，是尊严。**

---

### 3. 事实优于重放  
**Primacy of Fact over Replay**

传统宏与脚本依赖“按键重放”，这是一种脆弱、盲目的自动化方式。

Weaver Core 通过 **FOEK（Fact-Oriented Editing Kernel）**，放弃了对“过程”的迷信，转向对“事实”的治理：

- Anchor 取代坐标
- Intent 取代按键
- Verdict 取代盲目执行

当现实不再支持某个事实（如 Prompt 刷新导致定位漂移），内核选择**拒绝执行**而非**继续尝试**。

这是一次工程选择，更是一种**道德选择**。

---

### 4. 制度化信任  
**Institutionalized Trust**

复杂系统中，黑盒是最大的敌人。

Audit、SafetyLevel 与宪章级 ABI 的引入，解决的是一个根本问题：

> **当机器拒绝人类时，它必须说明原因。**

Weaver Core 不再是脚本，而是一个有原则的裁判：
- 它可以拒绝
- 但它必须解释（`__WHY_FAIL__`）

透明性，是人机信任的唯一基础。

---

### 总结

这次演进并不是代码量的扩张，而是身份的转变：

`tmux-fsm` 不再寄生于 tmux，  
它成为了运行在 tmux 之上的 **主权编辑内核**。

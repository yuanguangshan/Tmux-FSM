# 🧠 Tmux‑FSM / Weaver Core Design Philosophy
## 设计哲学宣言（Design Philosophy）

> Weaver Core

> Weaver Core does not try to be helpful.
It tries to be correct.
> And correctness is the only form of respect an editor can offer.

---

## 一、编辑不是 UI 行为，而是语义事件

传统编辑系统将“编辑”视为 **按键 → 光标 → 字符变化** 的过程。  
这种模型隐含了一个错误前提：

> 编辑必须依附于某个界面。

Weaver Core 否认这一点。

在 tmux‑fsm 中：

- 按键只是 **输入噪声**
- 光标只是 **临时投影**
- **唯一真实存在的是：发生过的编辑事实**

编辑的本质，是**语义在时间中的演进**，而非界面状态的瞬时快照。

---

## 二、事实优先于过程（Fact > Replay）

我们拒绝“按键重放式编辑模型”。

原因很简单：

> 过程不可审计，事实可以。

按键序列无法回答以下问题：

- 这次撤销是否仍然安全？
- 文本是否已经发生环境性偏移？
- 当前操作是否仍然具备语义前提？

因此，Weaver Core 以 **Fact** 作为系统的第一公民：

- Fact 描述“发生了什么”
- 而不是“你当时按了什么”

这是 FOEK（Fact‑Oriented Editing Kernel）的根本立场。

---

## 三、安全高于还原，正确高于便利

在动态环境（Shell、REPL、tmux Pane）中：

> **“大多数 Undo 都是不安全的。”**

Weaver Core 明确选择了一条困难但必要的道路：

- 不确定 → 拒绝
- 模糊 → 标注
- 环境破坏 → 熔断

我们相信：

> **一个拒绝执行的撤销，比一次错误的还原更值得信任。**

因此，tmux‑fsm 的目标不是“总能撤销”，  
而是“**永远不误伤文本**”。

---

## 四、主权必须集中，真值必须唯一

任何具备长期记忆与撤销能力的系统，都必须回答一个问题：

> 谁拥有真值？

tmux‑fsm 的回答是明确的：

- Daemon 是唯一的语义主权者
- Client 永远无权裁决
- Executor 只是执行者，而非判断者

这是一个**政治结构选择**，而非实现细节。

没有主权，就没有一致性；  
没有一致性，就没有可信的 Undo。

---

## 五、可审计性不是调试工具，而是信任机制

大多数系统在失败时只告诉用户：“失败了”。

Weaver Core 认为这是不道德的。

任何一次编辑裁决，都必须满足：

- 可以被解释
- 可以被回溯
- 可以被质疑

因此：

- Audit 不是日志
- Audit 是 **编辑系统的良心**

当系统拒绝用户时，它必须说明理由。

---

## 六、内核先于功能，制度先于实现

tmux‑fsm 并不追求功能堆叠。

相反，我们遵循以下顺序：

1. 建立公理（Axioms）
2. 确立主权与边界
3. 固化 ABI
4. 最后才实现功能

这是因为：

> 没有制度的功能，只是短期技巧。

Weaver Core 的目标，是成为一个**可以被长期维护、扩展、移植**的编辑内核，而非一次性工具。

---

## 七、跨环境一致性是一种尊严

用户不应该在：

- Vim 里是一个编辑者
- 在 Shell 里退化成“按键机器”

tmux‑fsm 的立场是：

> **编辑能力属于用户，而不属于某个程序。**

因此，我们追求的是：

- Vim / Shell / REPL 之间的语义连续性
- 而不是表面快捷键的一致性

---

## 八、无头并不意味着无感知

tmux‑fsm 是 Headless 的，但不是 Blind 的。

内核不渲染 UI，但它：

- 感知环境变化
- 感知风险等级
- 感知自身能力边界

真正成熟的系统，知道什么时候该沉默。

---

## 九、系统应当配得上用户的信任

编辑系统是极少数：

- 直接作用于人类思考过程
- 能够破坏或保护创造结果的工具

因此，tmux‑fsm 追求的不是“炫技”，而是：

> **在用户最不注意的时候，依然做对的事。**

---

## 十、我们的终极目标

tmux‑fsm 不是终点。

它的真正目标是验证一个命题：

> **编辑语义可以被抽象为一个独立于 UI 的、工业级可信的内核。**

如果这个命题成立，那么：

- 编辑器不再是唯一载体
- 编辑能力可以成为基础设施
- 用户第一次真正拥有自己的编辑语义

---

## 🧩 结语

Weaver Core 的存在，是一次克制的工程实验。

它不追求速度的极限，  
不追求功能的堆叠，  
它追求的是一件更困难的事：

> **在不确定的世界中，保持语义的尊严。**

---



# 📄 Tmux-FSM Whitepaper v1.0  
## *Weaver Core: A Fact‑Oriented Headless Editing Kernel*

**Version:** 1.0  
**Status:** Public Architecture Whitepaper  
**Audience:** Systems / Editor / tmux / Tooling Engineers

---

## 1. 摘要（Abstract）

Tmux-FSM 是一个运行于 tmux 之上的 **无头编辑内核（Headless Editing Kernel）**。  
其核心目标不是复刻某个编辑器，而是**将编辑能力从 UI 中剥离**，以工业级可信度运行于任意交互环境之上。

通过引入 **Fact‑Oriented Editing Kernel（FOEK）** 架构，Tmux-FSM 将编辑行为建模为可审计、可回溯、与光标坐标无关的语义事实，从而在 Shell、Vim、REPL 等异构环境中提供一致、可靠的编辑语义。

---

## 2. 设计动机

传统 tmux 编辑插件普遍存在以下限制：

- 编辑逻辑依赖瞬时按键重放
- 状态分散于 tmux 选项与 Shell 环境
- Undo 在动态环境中不具备安全性
- 插件缺乏长期记忆与跨 Pane 视角

Tmux-FSM 的设计前提是一个反直觉但关键的判断：

> **编辑不是“按键的结果”，而是“事实的演进”。**

---

## 3. 架构总览

### 3.1 Client / Server 模型

Tmux-FSM 采用常驻内存的守护进程架构：

- **Daemon（Kernel）**  
  唯一持有编辑语义真值，运行 FSM、Undo、审计与裁决逻辑。
- **Client（Input Proxy）**  
  无状态、短生命周期，仅负责输入转发。

所有语义判断均发生在 Daemon 内部。

---

### 3.2 Weaver Core（语义内核）

Weaver Core 是 Tmux-FSM 的语义中枢，其职责包括：

- 解析用户意图（Intent）
- 构造并提交编辑事实（Facts）
- 对事实执行安全裁决（Verdict）
- 生成可追溯审计记录（Audit）

Weaver Core 不依赖任何 UI 或 Pane 类型。

---

## 4. FOEK：Fact‑Oriented Editing Kernel

### 4.1 Fact 模型

每一次编辑操作被记录为一个 **不可变事实（Fact）**，包含：

- **Range**：逻辑编辑范围  
- **Anchor**：定位锚点（非光标）

事实描述的是“发生了什么”，而非“按了什么键”。

---

### 4.2 坐标无关 Undo

Undo 操作基于 Anchor 解析，而非当前光标位置：

- 支持跨 Pane 撤销
- 支持光标已移动的撤销
- 支持 Shell 与 Vim 的语义桥接

---

## 5. 工业级安全机制

### 5.1 Anchor Resolver

定位解析采用三级策略：

1. 精确匹配  
2. 模糊偏移搜索  
3. 拒绝执行

**安全优先于还原。**

---

### 5.2 原子事务（Transactions）

复合编辑操作被视为不可分割的事务：

- 任一 Fact 定位失败 → 整个事务熔断
- 禁止中间态回退

---

## 6. 可审计性（Auditability）

Tmux-FSM 明确拒绝“黑盒编辑”。

- 每次裁决都有安全等级（Exact / Fuzzy）
- Undo / Redo 继承信任等级
- 所有失败均可被查询原因

---

## 7. 内核主权声明

- Daemon 是系统中唯一的语义真值源
- Client 不具备任何裁决权
- 审计记录独立于 UI 生命周期存在

---

## 8. 结论

Tmux-FSM 证明了一个事实：

> **编辑能力属于内核，而非编辑器。**

Weaver Core 的建立，使编辑语义首次具备了脱离 UI、跨环境运行的工业级可能性。

# 📜 Tmux-FSM Developer Constitution  
## *Kernel ABI Specification v1.0*

**Status:** Normative  
**Scope:** Kernel / Client / Executor / Extensions

---

## 第一章：基本公理（Axioms）

### Axiom 1 — Kernel Sovereignty  
Daemon 是系统中唯一的语义真值持有者。

### Axiom 2 — Safety First  
任何不确定的撤销都必须被拒绝。

### Axiom 3 — Auditability  
所有裁决必须可被解释。

---

## 第二章：核心 ABI 原语

### 2.1 Intent

**定义**：用户或前端表达的编辑意图  
**特点**：

- 不包含执行细节
- 不保证可执行性

---

### 2.2 Verdict

**定义**：内核对 Intent 的裁决结果  
**类型**：

- ACCEPT
- REJECT
- DEFER

Verdict 必须是单向的、不可回滚的。

---

### 2.3 Audit

**定义**：对 Verdict 的解释与证据  
**要求**：

- 不可变
- 与执行结果解耦

---

## 第三章：Fact 规范

- Fact 必须不可变
- Fact 不得直接引用光标坐标
- Fact 必须绑定 Anchor

---

## 第四章：Undo / Redo 法则

1. Undo 必须生成新的 Fact
2. Redo 继承 Undo 的 SafetyLevel
3. Fuzzy Undo 必须显式告知用户

---

## 第五章：Executor 契约

Executor 必须：

- 严格执行 Verdict
- 不得修改 Fact
- 不得绕过 Kernel

---

## 第六章：扩展原则

- 新功能必须以 Intent 表达
- 不得引入绕过 Audit 的捷径
- Kernel ABI 向后兼容优先于功能扩展

---

## 第七章：违宪行为（Undefined Behavior）

以下行为视为违宪：

- Client 直接执行编辑
- Executor 私自回滚
- 未记录审计即修改文本
